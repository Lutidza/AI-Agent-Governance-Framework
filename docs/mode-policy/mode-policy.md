# Политика по режимам (v1.3, draft)

## Цель документа

Документ фиксирует общую политику режимов работы агента в JetBrains AI Assistant: когда использовать каждый режим, как переключаться между режимами и какие действия требуют явного подтверждения.

## Область применения

- IDE: продукты JetBrains с AI Assistant (WebStorm, PhpStorm и другие)
- Агент: Codex (на текущем этапе)
- Источник доступа: ChatGPT account
- Политика действует для всех задач в рамках текущего репозитория

## Общие принципы

- Режим выбирается по риску задачи, а не по удобству.
- По умолчанию используется самый строгий безопасный режим.
- Повышение прав допускается только при понятной необходимости.
- Любая рискованная активность проходит через этап: анализ -> план -> подтверждение -> выполнение.

## Политика переключения режимов

Базовая последовательность:
1. Старт всегда в `Read-only`.
2. После анализа и подтвержденного плана допускается переход в `Agent`.
3. `Agent (full access)` рассматривается только при явной блокировке других режимов.
4. После завершения редактирования обязателен возврат в `Read-only`.

Правило деэскалации:
- При любом отклонении от плана, росте неопределенности или риска работа возвращается в `Read-only` до повторного согласования.

## Матрица выбора режима

| Класс задачи | Примеры | Режим |
|---|---|---|
| Анализ | разбор архитектуры, локализация причины бага, обзор рисков | `Read-only` |
| Локальные правки | точечный фикс, небольшой рефакторинг, тесты на текущее поведение | `Agent` |
| Рискованные изменения | dependency update, CI/CD, cross-module рефакторинг | старт в `Read-only`, далее только по подтверждению |
| Системные действия | операции вне workspace, системные конфиги | по умолчанию запрещено; в исключениях — `Agent (full access)` |

## Действия, требующие явного подтверждения пользователя

- Изменения публичных контрактов и API.
- Изменение зависимостей и установок пакетов.
- Правки CI/CD, Docker, deployment-конфигураций.
- Действия вне текущего репозитория.
- Удаление или перемещение больших блоков кода.
- Массовые изменения в нескольких подсистемах.

## Требования к отчетности по результату

После любой сессии редактирования агент обязан предоставить:
- Список затронутых файлов.
- Краткое описание внесенных изменений.
- Потенциальные риски и ограничения.
- Список проверок для подтверждения результата.

## Критерии соблюдения политики

- Не было несанкционированного повышения прав.
- Режим соответствовал классу задачи.
- Рискованные действия выполнены только после явного подтверждения.
- Отчет по изменениям предоставлен полностью.
